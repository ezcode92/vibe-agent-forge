# Incremental Implementation

## 목적

큰 변경을 독립적으로 이해·구현·검증 가능한 단위로 나누고, 각 단계의 증거를 유지하면서 요청한 결과까지 진행한다. 이 skill은 사용자 요청의 범위나 권한을 확장하지 않는다.

## 사용 조건

- 현재 요청과 저장소 지침이 구현을 허용하는지 먼저 확인한다.
- 단일한 작은 변경이면 불필요하게 단계와 산출물을 늘리지 않는다.
- 단계 사이에 공개 계약, 데이터 또는 구조 결정이 있으면 선행 결정을 명시한다.

## 입력 Context

- 사용자 목표, 제외 범위와 완료 조건
- 적용되는 `AGENTS.md`, 관련 설계·설정·source·test
- 현재 Git 상태와 기존 사용자 변경
- 사용 가능한 build, test, lint와 검증 명령

## 작업 절차

1. 목표를 관찰 가능한 결과와 변경하지 않을 범위로 정리한다.
2. 현재 동작, 책임 경계, 의존 관계와 회귀 가능성을 확인한다.
3. 작업을 하나의 판단 또는 동작을 바꾸는 작은 단계로 나누고 각 단계의 검증을 정한다.
4. 가장 위험이 낮고 후속 작업의 근거가 되는 단계부터 최소 변경으로 구현한다.
5. 단계마다 가장 좁은 관련 검증을 실행하고 실패하면 다음 단계로 진행하지 않는다.
6. 중간 결과에 변경 내용, 검증, 새로 확인된 위험과 다음 단계를 요약한다.
7. 모든 단계를 결합한 뒤 필요한 범위의 회귀 검증과 최종 diff를 확인한다.

## 검증 기준

- 각 단계가 독립된 목적, 제한된 변경 범위와 확인 가능한 결과를 가진다.
- 단계별 검증과 최종 통합 검증 결과가 기록된다.
- 기존 공개 계약과 사용자 변경이 의도 없이 바뀌지 않는다.
- 실패한 단계의 원인과 수정 근거가 다음 변경에 반영된다.

## 금지 패턴

- 전체 구현을 한 번에 바꾼 뒤 마지막에만 검증하지 않는다.
- 단계 분리를 이유로 임시 abstraction, 호환되지 않는 중간 상태나 불필요한 파일을 남기지 않는다.
- 실패를 무시하거나 검증 범위를 줄여 성공으로 보고하지 않는다.
- 요청 범위 밖 리팩터링과 기능 추가를 후속 단계로 끼워 넣지 않는다.

## 완료 기준

- 모든 필수 단계와 사용자 완료 조건이 충족된다.
- 단계별·통합 검증 결과와 미실행 검증 사유가 정리된다.
- 최종 변경이 계획한 책임 경계에 한정되고 임시 작업이 남지 않는다.
- 남은 위험과 후속 작업이 완료 결과와 구분되어 보고된다.
