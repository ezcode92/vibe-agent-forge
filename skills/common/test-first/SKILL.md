# Test First

## 목적

구현 전에 기대 동작과 실패 조건을 검증 가능한 계약으로 만들고, 최소 구현과 회귀 확인으로 변경의 정확성을 증명한다. Test 추가가 불가능한 경우 동등한 검증 조건과 제한을 먼저 정의한다.

## 사용 조건

- 요청이 동작 변경 또는 회귀 방지를 포함하는지 확인한다.
- 문서·설정처럼 실행 test가 부적합하면 정적 검사나 명시적 review 기준을 사용한다.
- 기존 test 구조와 프로젝트 검증 관례를 우선한다.

## 입력 Context

- 사용자 요구, 기대 결과와 중요한 실패·경계 사례
- 관련 기존 test, fixture와 test 명령
- 변경 대상의 공개 계약과 현재 동작
- test 환경의 제약과 외부 dependency

## 작업 절차

1. 변경 전 기대 동작을 입력, 관찰 가능한 출력과 실패 조건으로 정의한다.
2. 기존 test가 이미 보장하는 범위와 누락된 사례를 확인한다.
3. 단위, contract, integration 또는 정적 검증 중 가장 좁고 신뢰할 수 있는 범위를 선택한다.
4. 변경 때문에 실패해야 하는 test 또는 검증 조건을 먼저 작성하고 예상 이유로 실패하는지 확인한다.
5. Test를 통과시키는 최소 변경을 구현하고 관련 없는 구조 개선은 분리한다.
6. 새 test, 기존 관련 test와 필요한 상위 검증을 순서대로 실행한다.
7. Test가 구현 세부사항보다 외부 동작과 중요한 경계를 검증하는지 다시 확인한다.

## 검증 기준

- 구현 전 실패가 요구사항과 직접 연결된 이유로 발생한다.
- 구현 후 새 검증과 기존 관련 검증이 모두 통과한다.
- 정상, 경계와 중요한 실패 경로가 필요한 수준으로 포함된다.
- Test가 비결정적 환경, 실행 순서와 실제 외부 service에 불필요하게 의존하지 않는다.

## 금지 패턴

- 구현 후 현재 결과를 그대로 승인하는 test만 추가하지 않는다.
- Coverage 수치만 높이기 위해 의미 없는 assertion을 만들지 않는다.
- Private 구현 구조에 과도하게 결합해 안전한 리팩터링을 막지 않는다.
- 실패 test를 skip하거나 기대값을 약화해 통과시키지 않는다.

## 완료 기준

- 기대 동작과 검증 범위가 test 또는 동등한 조건으로 고정된다.
- 최소 구현 후 관련 회귀 검증이 통과한다.
- 실행하지 못한 test와 환경 제약이 보고된다.
- 추가 test가 필요한 후속 위험이 완료 범위와 구분된다.
