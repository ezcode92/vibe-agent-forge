# Refactoring

## 목적

관찰 가능한 동작과 공개 계약을 보존하면서 명확한 구조 문제를 최소 단계로 개선한다. 새로운 기능, 정책 변경과 동작 수정은 별도 작업으로 분리한다.

## 사용 조건

- 개선할 구조 문제와 성공 기준을 먼저 확인한다.
- 동작 보존을 확인할 test 또는 대체 검증이 없으면 먼저 안전망을 마련한다.
- 현재 변경에 꼭 필요하지 않은 광범위한 정리는 수행하지 않는다.

## 입력 Context

- 리팩터링 목적과 변경하지 않을 동작
- 관련 source, 호출 관계, 공개 API와 데이터 계약
- 기존 test, 정적 분석과 성능 기준
- 현재 Git diff와 함께 진행 중인 기능 변경

## 작업 절차

1. 중복, 긴 책임, 잘못된 dependency 등 구체적인 개선 대상을 정의한다.
2. 현재 동작과 공개 계약을 test 또는 관찰 가능한 조건으로 고정한다.
3. 이름 변경, 책임 이동, 추출 등 되돌리기 쉬운 작은 단계로 계획한다.
4. 한 단계만 적용하고 관련 test와 정적 검증으로 동작 보존을 확인한다.
5. 실패하면 해당 단계의 가정을 재검토하고 다음 구조 변경을 섞지 않는다.
6. Formatting-only 변경은 기능·구조 diff와 분리해 review 노이즈를 제한한다.
7. 전체 diff에서 불필요한 abstraction과 의도하지 않은 계약 변경을 확인한다.

## 검증 기준

- Refactoring 전후 동일한 입력에 대한 외부 동작과 실패 의미가 유지된다.
- Public API, serialization, 데이터와 설정 계약의 의도치 않은 변경이 없다.
- 각 단계 후 관련 test와 필요한 상위 검증이 통과한다.
- 결과 구조가 원래 문제를 줄이고 새로운 간접성이나 결합을 만들지 않는다.

## 금지 패턴

- Refactoring에 기능 추가, 버그 수정과 정책 변경을 숨겨 섞지 않는다.
- Test 없이 대규모 이동, 이름 변경과 abstraction 교체를 한 번에 수행하지 않는다.
- 패턴 적용이나 중복 제거 자체를 목적으로 새 계층을 만들지 않는다.
- Formatting-only 변경으로 의미 있는 diff를 덮지 않는다.

## 완료 기준

- 명시한 구조 문제가 줄고 동작 보존 근거가 있다.
- 작은 단계와 검증 결과를 추적할 수 있다.
- 기능·formatting 변경이 분리되거나 불가피한 경우 이유가 설명된다.
- 남은 구조 문제는 현재 범위와 구분해 기록된다.
