# Debugging

## 목적

증상과 근본 원인을 구분하고 재현 가능한 증거에 기반해 최소 범위로 문제를 해결한다. 진단만 요청된 경우 원인과 수정 방향까지만 제시하고 코드를 변경하지 않는다.

## 사용 조건

- 기대 동작과 실제 동작의 차이를 확인할 수 있어야 한다.
- 운영 환경 변경이나 destructive 진단이 필요하면 현재 권한을 확인한다.
- 재현할 수 없으면 추정 수정 대신 필요한 관측 정보와 제한을 명시한다.

## 입력 Context

- 재현 절차, 입력, 환경과 기대·실제 결과
- 오류 message, stack trace, 관련 log와 최근 변경
- 관련 source, 설정, test와 dependency 경계
- 이미 시도한 우회·수정과 그 결과

## 작업 절차

1. 증상을 가장 작은 입력과 환경에서 재현하고 실패 시점을 고정한다.
2. Log와 오류를 원문 기준으로 확인하되 secret과 개인정보는 노출하지 않는다.
3. 정상 경로와 실패 경로의 차이를 좁혀 가능한 원인 가설을 세운다.
4. 가설마다 관찰되거나 반증될 결과를 정하고 한 변수씩 확인한다.
5. 확인된 근본 원인을 해결하는 최소 수정을 선택한다. 수정 권한이 없으면 제안에 그친다.
6. 원래 재현 절차, 경계·실패 사례와 관련 회귀 test를 다시 실행한다.
7. 임시 우회가 있다면 적용 조건, 위험, 제거 기준과 근본 해결을 분리해 기록한다.

## 검증 기준

- 수정 전 실패가 재현되고 수정 후 같은 조건에서 성공한다.
- 가설과 결론이 log, test 또는 코드 경로의 증거로 연결된다.
- 관련 정상 경로와 인접 실패 경로에 회귀가 없다.
- 환경 차이나 비결정성이 결과에 미치는 영향을 확인했다.

## 금지 패턴

- 재현과 증거 없이 가능성만으로 코드를 변경하지 않는다.
- 오류를 catch·무시하거나 검증을 비활성화해 증상만 숨기지 않는다.
- 여러 가설의 수정을 한 번에 적용해 원인을 판별할 수 없게 하지 않는다.
- 임시 우회를 근본 해결로 보고하거나 제거 조건 없이 남기지 않는다.

## 완료 기준

- 근본 원인 또는 진단 불가의 구체적인 제한이 설명된다.
- 허용된 경우 최소 수정이 적용되고 재현·회귀 검증이 통과한다.
- 임시 우회와 근본 해결이 명확히 구분된다.
- 미검증 환경과 남은 위험이 최종 결과에 포함된다.
