# Code Review

## 목적

변경이 의도한 문제를 안전하게 해결하는지 증거 중심으로 검토하고, 실제 수정 가치가 있는 finding을 영향도 순으로 보고한다. 별도 요청이 없으면 review 대상 코드를 수정하지 않는다.

## 사용 조건

- Review할 diff, 변경 파일 또는 비교 기준이 식별되어야 한다.
- 스타일 취향보다 correctness, 회귀와 유지보수 위험을 우선한다.
- 변경 범위 밖 기존 문제는 이번 변경으로 악화되거나 직접 관련될 때만 포함한다.

## 입력 Context

- 변경 목적, 요구사항과 완료 조건
- Diff, 관련 source·test·설정과 공개 계약
- 적용되는 architecture, 보안과 coding 지침
- 실행 가능한 검증 명령 및 기존 검증 결과

## 작업 절차

1. 변경 의도와 실제 diff 범위를 비교하고 누락·범위 확장을 확인한다.
2. 호출자, 데이터, 설정과 외부 계약에 미치는 영향을 추적한다.
3. 정상·경계·실패 흐름의 correctness와 회귀 가능성을 검토한다.
4. 책임, dependency, testability, 불필요한 abstraction과 중복을 검토한다.
5. 입력, 권한, secret, injection과 민감정보 노출 위험을 확인한다.
6. Test가 변경 동작과 실패 경로를 실제로 검증하는지 확인하고 가능한 좁은 검증을 실행한다.
7. Finding별 위치, 재현 조건, 영향과 최소 수정 방향을 심각도 순으로 정리한다.

## 검증 기준

- 각 finding은 구체적인 코드 위치와 실패 또는 유지보수 영향에 연결된다.
- 변경으로 발생한 문제와 기존 문제를 구분한다.
- Test 및 문서 누락은 보호해야 할 동작과 연결해 설명한다.
- Finding이 없으면 검토 범위, 실행한 검증과 남은 위험을 명시한다.

## 금지 패턴

- 근거 없는 가능성, 개인 취향과 formatting만을 결함으로 보고하지 않는다.
- Diff를 읽지 않고 일반 체크리스트 결과만 나열하지 않는다.
- 동일 근본 원인을 여러 finding으로 중복 보고하지 않는다.
- 변경 요청이 없는 review에서 임의로 코드를 수정하거나 범위를 확장하지 않는다.

## 완료 기준

- 실행 가능한 finding이 심각도와 근거를 포함해 정리된다.
- 정확성, 설계, 보안과 test 관점의 검토 범위가 명확하다.
- 실행한 검증과 실행하지 못한 검증이 구분된다.
- 잔여 위험과 확인이 필요한 가정이 보고된다.
