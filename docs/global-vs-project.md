# 전역 설정과 프로젝트 설정의 책임 분리

## 분리 원칙

설정의 위치는 “얼마나 자주 쓰이는가”가 아니라 “어떤 범위에서 항상 참이어야 하는가”로 결정한다. 모든 저장소에 적용할 수 있는 원칙만 전역 설정에 두고, 저장소의 목표나 도구에 의존하는 규칙은 프로젝트 설정에 둔다.

## `~/.codex/AGENTS.md`의 책임

전역 `AGENTS.md`에는 다음처럼 프로젝트가 바뀌어도 유지되는 작업 원칙을 둔다.

- 기본 응답 언어와 커뮤니케이션 방식
- context를 좁혀 읽는 순서와 사실 확인 원칙
- 최소 변경, 기존 변경 보호, 불필요한 리팩터링 금지
- 공통 검증, 보안, secret 취급, destructive command 제한
- Git 상태 확인, 관련 파일만 staging하는 공통 Git 원칙
- skill 선택 기준과 저장소별 지침 우선 원칙

특정 빌드 명령, 패키지 구조, 프레임워크 버전, API·DB 규칙, 배포 환경은 전역 설정에 넣지 않는다.

## 저장소 `AGENTS.md`의 책임

저장소 `AGENTS.md`에는 해당 저장소에서 작업할 때 항상 필요한 정보만 둔다.

- 프로젝트 목적과 현재 개발 Phase
- 허용하거나 금지하는 변경 범위
- 저장소의 디렉터리 책임과 아키텍처 경계
- 실제 build, test, lint, format 명령
- 프로젝트에 고유한 API, DB, 보안, 배포 제약
- 변경 시 함께 갱신해야 하는 문서나 산출물
- 프로젝트 전용 skill과 자동화의 목적 및 사용 제한

AgentForge의 현재 저장소 지침에는 문서 중심 설계 단계, 미구현 범위, fragment·skill 분리 원칙과 git-auto hook의 예외적 존재가 포함된다.

## 다른 구성 요소로 분리할 내용

| 위치 | 분리 기준 | 예시 |
| --- | --- | --- |
| skill | 특정 작업에서만 필요한 순서, 체크리스트, 검증·완료 기준 | 디버깅, 테스트 보강, 보안 검토 |
| docs | 작업 지시가 아니라 배경, 설계 결정, 계약을 설명하는 내용 | profile 명세, merge 정책, 아키텍처 결정 |
| stack fragment | 하나의 스택을 선택했을 때 항상 필요한 짧은 실행 지침 | Kotlin 관례, React 컴포넌트 원칙 |
| bridge fragment | 두 개 이상의 스택 조합에서만 성립하는 규칙 | Spring REST API와 React client 간 계약 |

긴 설명을 줄이기 위해 skill이나 docs로 분리하더라도, `AGENTS.md`에는 해당 자료를 언제 적용해야 하는지 찾을 수 있는 짧은 진입 규칙을 남긴다. fragment에는 독립 스택의 책임만 두며 다른 스택을 전제하지 않는다.

## 충돌 시 우선순위

실제 작업 지침이 충돌하면 다음 순서로 판단한다.

1. 사용자의 현재 요청
2. 현재 작업 경로에 가장 가까운 하위 `AGENTS.md` 또는 override 지침
3. 저장소 루트 `AGENTS.md`
4. 저장소의 설계 문서와 프로젝트 설정
5. 전역 `~/.codex/AGENTS.md`

skill은 호출된 작업 범위에서 절차를 구체화하지만 상위 지침의 허용 범위를 넓히지 않는다. stack·bridge fragment의 내부 병합 우선순위는 `merge-policy.md`를 따르며, 병합 결과도 위의 작업 지침 우선순위를 넘지 않는다. 충돌을 자동으로 해소할 근거가 없으면 임의 선택하지 않고 충돌로 보고한다.
