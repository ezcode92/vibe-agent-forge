# Fragment Merge 정책

## 기본 원칙

Fragment merge는 여러 Markdown 파일을 단순 concat하는 작업이 아니다. 각 fragment를 의미 있는 section과 규칙 단위로 식별하고, 우선순위·중복·충돌을 검증해 하나의 간결한 최종 지침으로 구성해야 한다.

단순 concat을 금지하는 이유는 다음과 같다.

- 동일 원칙이 표현만 달리해 반복될 수 있다.
- 서로 반대되는 지침이 모두 남아 에이전트가 임의 해석하게 된다.
- section 구조와 참조 관계가 깨질 수 있다.
- 최종 `AGENTS.md`가 불필요하게 커져 context를 낭비한다.

## Merge 단위

각 규칙은 최소한 출처, 적용 조건, 의미 영역, 우선순위를 추적할 수 있어야 한다. 병합은 heading 문자열만이 아니라 규칙의 대상과 효과를 기준으로 수행한다. 출처 정보는 충돌 진단에 사용하되 최종 문서에 반복 출력하지 않는다.

## Merge priority

낮은 우선순위에서 높은 우선순위 순서는 다음과 같다.

1. 공통 quality·기본 원칙
2. language fragment
3. architecture, database, API fragment
4. framework/platform fragment
5. bridge fragment
6. profile의 명시적 override
7. 저장소 project 지침과 사용자 현재 요청

높은 우선순위는 낮은 우선순위를 무조건 삭제하는 권한이 아니다. 같은 의미 영역에서 더 구체적인 적용 조건을 가진 규칙만 대체할 수 있다. project 지침과 사용자 요청은 생성 결과 밖에서도 최종 권한을 가지며, fragment merge가 그 범위를 확장하거나 약화할 수 없다.

## 중복 제거 기준

- 대상, 조건, 요구 동작이 같은 규칙은 하나만 유지한다.
- 일반 규칙과 구체 규칙이 양립하면 일반 규칙을 한 번만 두고 구체적인 차이만 남긴다.
- 같은 용어의 설명은 최초 정의 위치에 두고 다른 section에서는 참조한다.
- 단어가 같아도 적용 범위나 강도가 다르면 자동으로 중복 처리하지 않는다.
- 중복 제거 후에도 어떤 source가 기여했는지 진단 정보로 추적할 수 있어야 한다.

## 충돌 감지 기준

다음 경우는 충돌로 본다.

- 같은 조건에서 하나는 요구하고 다른 하나는 금지하는 경우
- 동일한 책임을 서로 다른 계층이나 구성 요소에 배정하는 경우
- 양립할 수 없는 도구, architecture 또는 lifecycle을 동시에 필수로 지정하는 경우
- 상위 지침의 금지 범위를 낮은 우선순위 fragment가 허용하는 경우
- 우선순위와 구체성만으로 어느 규칙을 유지할지 결정할 수 없는 경우

우선순위로 안전하게 해결할 수 있는 충돌은 대체된 source와 이유를 진단에 남긴다. 해결 근거가 없으면 생성에 실패해야 하며 임의로 한쪽을 선택하거나 두 규칙을 모두 출력하지 않는다.

## Bridge fragment

Bridge fragment는 독립적으로 재사용 가능한 stack fragment를 오염시키지 않으면서 조합 경계를 설명한다. 예를 들어 Spring과 React의 개별 fragment에는 각 생태계 규칙만 두고, request·response schema 공유나 error contract 같은 연결 규칙은 Spring–React bridge가 담당한다.

Bridge는 다음 조건에서만 추가한다.

- 둘 이상의 선택 항목이 동시에 있을 때만 규칙이 유효하다.
- 규칙을 한쪽 fragment에 넣으면 반대쪽 기술을 불필요하게 전제하게 된다.
- 단순 중복 제거가 아니라 실제 interface 또는 lifecycle 조정이 필요하다.

## 최종 `AGENTS.md` 크기 제한

- 최종 문서에는 모든 작업에서 항상 필요한 실행 지침만 포함한다.
- 긴 배경 설명은 docs로, 조건부 절차는 skill로 이동한다.
- 같은 의미의 규칙을 여러 section에 반복하지 않는다.
- profile은 출력 크기 예산과 필수 section을 선언할 수 있어야 한다.
- 예산 초과 시 글자를 임의 절단하지 않고 중복, 설명, 조건부 절차 순으로 분리 대상을 진단한다.
- 필수 규칙을 유지한 상태에서 예산을 만족하지 못하면 생성 실패로 처리한다.

구체적인 기본 크기 수치는 실제 fragment 표본과 대상 에이전트의 context 비용을 측정하는 Phase 1에서 결정한다.
