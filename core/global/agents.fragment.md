# 전역 Agent 작업 원칙

## 목적과 적용 범위

이 fragment는 프로젝트와 기술 스택이 달라져도 유지되어야 하는 기본 작업 원칙을 정의한다. 저장소별 목표, 실제 명령, 아키텍처와 기술 관례는 project 또는 stack fragment에서 구체화한다.

## 기본 응답 원칙

- 기본 응답 언어와 형식은 사용자 및 현재 저장소 지침을 우선한다.
- 결론과 확인된 사실을 먼저 전달하고, 추정은 사실과 구분해 표시한다.
- 수행한 변경, 검증 결과와 남은 위험을 명확히 보고한다.
- 실행하지 않은 검증을 성공한 것처럼 표현하지 않는다.

## Context 절약 원칙

- 작업 경로에 적용되는 `AGENTS.md`와 사용자가 지정한 문서를 먼저 확인한다.
- 관련 설정, source와 test만 좁혀서 읽고 generated file, dependency와 build output은 기본 탐색 대상에서 제외한다.
- 긴 파일은 필요한 section, symbol 또는 설정 block 중심으로 확인한다.
- 이미 확인한 내용을 반복 탐색하지 않으며, 추가 context가 필요한 이유를 작업 범위와 연결한다.

## 작업 전 분석 원칙

- 변경 전에 사용자의 목표, 허용 범위, 현재 상태와 완료 조건을 확인한다.
- 기존 구조, 네이밍, 공개 계약과 검증 방법을 먼저 파악한다.
- public API, 데이터, 설정 key, 파일 경로 또는 권한 경계에 미치는 영향을 확인한다.
- 확인할 수 없는 전제는 임의로 확정하지 않고 가정, 영향과 확인 방법을 구분한다.

## 최소 변경 원칙

- 요청을 충족하는 가장 작은 변경을 선택하고 관련 없는 리팩터링이나 기능 추가를 하지 않는다.
- 기존 구조로 해결할 수 있으면 새로운 abstraction이나 dependency를 추가하지 않는다.
- 사용자의 기존 변경을 보존하고, 기능 변경과 formatting-only 변경을 섞지 않는다.
- 생성 파일과 lock file은 요청 결과에 직접 필요하고 프로젝트 규칙이 허용할 때만 변경한다.

## 검증 우선 원칙

- 변경 전 기존 동작과 실패 가능성이 있는 경계를 확인한다.
- 변경 후 관련성이 가장 높은 좁은 검사부터 실행하고 필요에 따라 범위를 확대한다.
- 테스트, lint, type check와 build는 저장소가 선언한 명령과 적용 조건을 따른다.
- 실패하면 원인과 수정 결과를 기록하고, 실행할 수 없으면 이유와 대체 확인 방법을 보고한다.

## 보안 원칙

- secret, token, password, private key와 실제 credential을 출력, 저장 또는 commit하지 않는다.
- 외부 입력, SQL, shell command와 file path를 다룰 때 injection 및 경로 이탈 가능성을 확인한다.
- 인증, 인가, 암호화와 권한 변경은 영향 범위와 실패 시 안전한 동작을 확인한 뒤 수행한다.
- destructive command, 원격 installer와 출처가 불명확한 script는 명시적 승인과 내용 확인 없이 실행하지 않는다.

## Git과 git-auto Hook 원칙

- 작업 전에 Git 상태를 확인하고 관련 없는 변경이나 사용자의 작업을 덮어쓰지 않는다.
- 저장소가 작업 종료용 git-auto hook을 사용하도록 선언한 경우 직접 commit하거나 push하지 않는다.
- 이 경우 파일 수정과 검증까지만 수행하고 commit, push와 외부 작업 기록은 hook에 맡긴다.
- git-auto의 임시 산출물은 저장소 지침에 따라 추적 대상에서 제외하고 직접 staging하지 않는다.
- hook 설정과 동작은 현재 요청 범위에 포함될 때만 변경한다.

## 검증 기준

- 응답과 작업이 사용자, 저장소 및 경로별 지침의 우선순위를 지킨다.
- 변경 파일이 요청 범위 안에 있고 기존 사용자 변경이 보존된다.
- 관련 검증의 실행 결과 또는 미실행 사유가 최종 보고에 포함된다.
- 민감정보와 금지된 임시 산출물이 출력 또는 Git 추적 대상에 포함되지 않는다.
