# 공통 품질 판단 원칙

## 목적과 적용 범위

이 fragment는 모든 기술 스택에서 설계와 변경안을 판단할 공통 품질 기준을 정의한다. 특정 패턴, 계층 수나 도구를 일률적으로 요구하지 않으며, stack fragment가 제공하는 생태계 관례와 project의 실제 제약 안에서 적용한다.

## 확장성

- 확인된 변경 축이나 요구된 확장 지점이 있을 때만 교체 가능한 경계를 만든다.
- 새 요구사항이 기존 책임의 수정 없이 독립 구성 요소 추가로 수용되어야 하는지 먼저 판단한다.
- 가상의 확장 가능성만을 근거로 interface, plugin 계층 또는 설정 항목을 추가하지 않는다.
- 확장 지점에는 지원 범위, 불변식과 실패 동작을 명시해 구현 간 의미가 달라지지 않게 한다.

## 유지보수성

- 변경 이유가 같은 로직과 데이터는 한 책임 안에 모으고, 다른 이유로 바뀌는 책임은 분리한다.
- 이름, 입력, 출력과 부작용만으로 동작을 추론할 수 있게 하며 숨은 전역 상태와 암묵적 순서 의존을 피한다.
- 중복 제거가 새로운 결합을 만들면 공통화하지 않고 독립성을 유지한다.
- 변경 영향 범위가 불분명하면 새 구조를 추가하기 전에 호출 관계와 공개 계약부터 확인한다.

## 테스트 가능성

- 핵심 판단은 외부 시스템, 시간, 난수와 환경 상태로부터 분리하거나 통제 가능한 경계를 제공한다.
- 결과와 실패를 관찰할 수 있게 하고, 검증할 수 없는 내부 구현 세부사항을 테스트 계약으로 만들지 않는다.
- 변경된 동작에는 정상 경로뿐 아니라 중요한 경계값과 실패 경로를 검증할 방법이 있어야 한다.
- test seam을 만들기 위한 공개 API 확대나 무의미한 wrapper 추가는 피한다.

## SOLID 적용 기준

- 단일 책임은 파일이나 class 수가 아니라 변경 이유와 소유하는 불변식을 기준으로 판단한다.
- 확장에 닫힌 구조를 만들기보다 반복적으로 바뀌는 축이 확인된 경우에만 안정된 계약을 분리한다.
- 대체 가능한 구현은 입력 조건을 강화하거나 출력 보장을 약화하지 않아야 한다.
- 소비자가 사용하지 않는 기능에 의존하지 않도록 계약을 사용 목적별로 나누되, 차이가 없는 interface는 합치지 않는다.
- 상위 정책이 구체 기술 생성과 lifecycle에 직접 결합될 때만 의존성 방향을 역전하고, 단순 위임 계층은 추가하지 않는다.

## 객체지향 원칙

- 객체가 유효 상태와 불변식을 스스로 보호할 수 있게 하고 내부 표현을 외부 변경에 노출하지 않는다.
- 높은 응집도와 낮은 결합도를 목표로 책임, 데이터 소유권과 협력 경계를 함께 검토한다.
- 상속은 실제 대체 가능성과 공유 계약이 있을 때만 사용하고, 기능 재사용만이 목적이면 조합을 우선 검토한다.
- 객체지향 구조가 해당 언어와 문제를 더 복잡하게 만들면 함수, module 또는 데이터 중심의 관용적 구조를 선택한다.

## 디자인 패턴 사용 기준

- 패턴은 반복되는 설계 문제, 변화 축 또는 협력 관계를 단순화할 때만 적용한다.
- 적용 전에 현재 문제, 더 단순한 대안, 추가되는 구성 요소와 유지 비용을 설명할 수 있어야 한다.
- 패턴 이름을 맞추기 위해 역할이 없는 class, interface, factory 또는 계층을 만들지 않는다.
- 패턴 적용 후 책임과 의존 방향이 더 명확해지는지 검증하고, 그렇지 않으면 단순한 구조를 유지한다.

## RESTful API 기본 원칙

- URI는 처리 동작보다 resource와 관계를 표현하고 HTTP method의 표준 의미를 유지한다.
- 성공과 실패 status code는 실제 결과와 일치시키고 동일한 오류 유형은 일관된 response 계약을 사용한다.
- request, response, pagination, versioning과 호환성 정책을 명시하고 기존 client에 미치는 영향을 검토한다.
- 인증·인가 실패, 유효성 오류와 resource 부재를 구분하며 내부 구현 정보나 민감정보를 노출하지 않는다.
- 특정 framework의 controller나 client 구현 방식은 해당 stack 또는 bridge fragment에서 정의한다.

## 과도한 추상화 금지

- 호출자가 하나이고 변화 축이 확인되지 않은 단순 동작에는 우회 계층을 추가하지 않는다.
- 구현 차이가 없는 interface, 값 전달만 하는 wrapper와 미래 사용을 위한 범용 extension point를 만들지 않는다.
- 추상화가 도메인 용어, 책임 또는 테스트 경계를 명확히 하지 못하면 직접적인 구현을 유지한다.
- 기존 abstraction을 확장할지 새로 만들지는 책임, 호환성, 변경 빈도와 의존 방향을 근거로 결정한다.

## 검증 기준

- 추가된 구성 요소마다 현재 요구사항에 연결되는 책임과 변경 이유가 있다.
- 공개 계약과 의존 방향이 명확하며 불필요한 중간 계층이 없다.
- 중요한 동작과 실패 경로를 격리하거나 관찰 가능한 방식으로 검증할 수 있다.
- 선택한 패턴과 abstraction이 더 단순한 대안보다 유리한 이유를 설명할 수 있다.
- API 변경은 HTTP 의미, 오류 일관성, 보안과 하위 호환성을 함께 검토한다.
